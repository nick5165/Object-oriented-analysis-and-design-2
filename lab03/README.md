# 🏡 Отчёт по проекту: Система управления «Умный дом»

## 📋 О проекте

Данный проект представляет собой программную симуляцию экосистемы **«Умного дома»** с графическим интерфейсом на базе фреймворка `PyQt6`. 
Пользователь может управлять различными домашними устройствами через виртуальный пульт, отслеживать их состояние в реальном времени, а также пользоваться встроенным медиаплеером. 

Ключевой особенностью системы является **механизм истории и отмены действий (Undo)**, который позволяет вернуть устройства в предыдущее состояние (например, отменить случайное включение пылесоса или сбросить громкость колонки).

### 🔌 Интегрированные устройства:
1. **💡 Умная лампа:** Управление питанием, цветом, яркостью и специальными режимами (Relax Mode).
2. **🔋 Умная розетка:** Базовое реле контроля питания.
3. **🎵 Умная колонка:** Полноценный медиаплеер (воспроизведение .mp3, управление громкостью, переключение треков).
4. **🤖 Робот-пылесос:** Управление уборкой, переключение режимов мощности (Silent, Standard, Turbo) и возврат на док-станцию.

## 🏗 Архитектура приложения

Проект разделен на независимые модули, что обеспечивает высокую масштабируемость:

* **`core/`** — Абстрактные базовые классы (`Device`, `Command`, `History`). Они задают "правила игры" для всей системы.
* **`app/`** — Ядро приложения (`Application`), связывающее графический интерфейс с логикой устройств через историю команд.
* **`devices/`** — Конкретные реализации устройств. Каждое устройство полностью инкапсулирует свое внутреннее состояние.
* **`gui/`** — Слой представления. Содержит главное окно и кастомные виджеты для визуализации (отрисовка волн динамика, движение пылесоса).

## ⚔️ Выбор архитектуры: Паттерн «Команда» vs Прямое управление

В процессе разработки был проведен архитектурный эксперимент: проект был реализован в двух вариациях.
1. **Прямое управление (Без паттернов):** Графический интерфейс вызывает методы устройств напрямую (через `lambda`-функции) и сохраняет "слепки" состояний в стек для отмены.
2. **С использованием паттерна «Команда» (Command):** Каждое действие оборачивается в независимый объект класса-наследника `Command`.

**Первичное наблюдение:** Версия без паттерна получилась короче по количеству строк кода. На первый взгляд она кажется проще. Однако, при детальном анализе масштабируемости проекта, архитектура "прямого вызова" продемонстрировала критические уязвимости. 

Ниже приведены **5 главных аргументов**, доказывающих, почему использование паттерна «Команда» является единственно верным решением для проектов такого уровня.

### 🛡 Аргумент 1. Соблюдение принципа SRP (Single Responsibility)
Без паттерна Команда графический интерфейс (GUI) вынужден знать всю внутреннюю логику устройств. 
Например, чтобы включить режим "Relax" у лампы, кнопке в GUI нужно знать, что яркость должна быть 30, а цвет — желтым. 
* **Проблема:** Если производитель лампы изменит API, придется переписывать код графического интерфейса.
* **Решение (Команда):** GUI просто отправляет объект `RelaxModeCommand`. Кнопка не знает, *как именно* лампа расслабляет пользователя. Код интерфейса остается чистым и независимым.

### 🚀 Аргумент 2. Макросы и мульти-команды (Киллер-фича)
В умном доме часто требуются комплексные сценарии, например, кнопка **«Я ухожу из дома»**. 
Она должна: *выключить свет + отключить розетки + выключить музыку + запустить пылесос*.
* **Без паттерна:** Потребуется написать огромную спагетти-функцию прямо в интерфейсе и придумать сложнейший механизм сохранения 4 разных бэкапов для возможной отмены.
* **С паттерном:** Создается класс `MacroCommand`, который просто принимает массив других команд и выполняет их в цикле. Отмена (`undo`) такого макроса работает «из коробки» (команды отменяются в обратном порядке).

### ⏱ Аргумент 3. Отложенное выполнение, таймеры и очереди
Функции умного дома подразумевают отложенные действия (например, *"Выключи свет через 30 минут"*).
* Функция (лямбда) — это просто инструкция в оперативной памяти. Её крайне сложно сериализовать (превратить в JSON), сохранить в базу данных или передать по сети с телефона на домашний сервер.
* Объект `Command` содержит все необходимые данные внутри себя. Его можно легко положить в очередь (Task Queue), передать в другой поток или отправить по сети.

### 📝 Аргумент 4. Логирование и прозрачная аналитика
Для отладки и статистики необходимо понимать, что происходит в доме.
В архитектуре без команд в лог будут сыпаться записи вида: `Выполнена функция <lambda> at 0x000...`. Это абсолютно неинформативно.
Паттерн «Команда» позволяет базовому классу выводить имя самого объекта:
```text
[LOG] Выполнена команда: StartCleaningCommand (Устройство: Пылесос)
[LOG] Выполнена команда: ChangeVolumeCommand (Устройство: Колонка, Значение: 80)
```
Это делает отладку прозрачной и приятной.

### 🔄 Аргумент 5. Сложная логика отмены (Undo)
В упрощенной версии отмена действий работала через слепое восстановление "словаря состояния" (Dict). Для лампочки это работает отлично. 
Но представим интеграцию **«Умной кормушки для животных»**. Действие: *«Насыпать 50г корма»*. 
Вы не можете сделать Undo этого действия, просто вернув словарю значение `food_dispensed = False`. Корм уже высыпан!
* Паттерн Команда позволяет написать для такого случая **уникальный** метод `undo()`. Например, отправить уведомление: `"Внимание: корм был высыпан, отмена физически невозможна"`. Объектный подход даёт гибкость, недоступную при банальном копировании состояния.

## 💻 Сравнение кода (Интерфейс)

Визуальное подтверждение того, как паттерн очищает логику приложения:

❌ **Плохо (Прямой вызов в GUI):**
```python
# GUI знает параметры, методы и логику устройства. Сильная связность.
def on_relax_mode_pressed(self):
    def relax_logic():
        self.lamp.set_brightness(30)
        self.lamp.set_color("Warm Yellow")
        self.lamp.turn_on()
    self.app.execute(self.lamp, relax_logic)
```

✅ **Отлично (Паттерн Команда):**
```python
# GUI просто делегирует задачу. Слабая связность.
def on_relax_mode_pressed(self):
    self.app.execute_command(RelaxModeCommand(self.app, self.lamp))
```

## 🎯 Заключение

Эксперимент доказал важное правило инженерии: **«Меньше строк кода — не всегда означает лучшую архитектуру»**. 

Отказ от паттернов проектирования в пользу быстрых решений (лямбда-вызовов) может дать краткосрочный выигрыш во времени разработки. Однако при появлении новых требований (голосовое управление, интеграция с сервером, сценарии автоматизации) архитектура прямого вызова рухнет под собственным весом.

Использование паттерна **«Команда»** в проекте «Умный дом» обеспечило:
* Соблюдение SOLID принципов.
* Идеальную инкапсуляцию устройств.
* Безопасный и расширяемый механизм истории (Undo/Redo).
* Легкость добавления новых устройств и команд без модификации старого кода.
