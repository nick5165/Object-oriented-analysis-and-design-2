# Отчет по лабораторной работе: Реализация паттерна «Легковес» (Flyweight) в системе высокопроизводительного мониторинга логов

## 1. Введение и постановка задачи
Целью данной работы является проектирование и реализация системы визуализации логов, способной эффективно обрабатывать миллионы записей в режиме реального времени. 

**Проблема:** В типичных системах мониторинга каждая строка лога содержит не только текст сообщения, но и метаданные визуализации (название шрифта, размер, цвет, начертание). При наивном подходе создание миллионов таких объектов приводит к катастрофическому потреблению оперативной памяти (RAM), так как параметры стиля дублируются для каждой строки.

**Решение:** Применение паттерна **Flyweight (Легковес)** для разделения внутреннего состояния (общие параметры стиля) и внешнего состояния (текст сообщения).

---

## 2. Архитектура системы
Система построена на принципах инверсии зависимостей (DIP). В основе лежит абстрактный интерфейс `ILogEngine`, что позволяет переключать движки хранения данных «на лету» без изменения кода графического интерфейса.

### 2.1. Основные компоненты:
1.  **ILogEngine (API):** Абстрактный интерфейс, определяющий методы добавления логов, получения доступа к данным и сбора телеметрии (память, количество объектов).
2.  **NaiveEngine (Без паттерна):** Реализация, хранящая в каждой записи полный набор данных стиля. Демонстрирует неэффективное использование ресурсов.
3.  **FlyweightEngine (Паттерн Легковес):** Реализация, использующая общие объекты стилей.
4.  **GUI (Dear ImGui):** Графическая оболочка с системой виртуального скроллинга и визуализацией графиков потребления RAM.

---

## 3. Реализация паттерна Легковес (Flyweight)

В проекте паттерн реализован через разделение данных на две структуры:

### 3.1. Внутреннее состояние (Intrinsic State) — `LogStyle`
Эта структура содержит общие свойства, которые повторяются у большинства логов (название шрифта "Consolas", размер 12, цвета RGB, флаги Bold/Italic). Эти объекты создаются в единственном экземпляре для каждого типа лога (`Info`, `Warning`, `Error`, `Critical`).

### 3.2. Внешнее состояние (Extrinsic State) — `FlyWeightEntry`
Объект, который остается уникальным для каждой записи. Он содержит только:
*   Уникальный текст сообщения (`std::string`).
*   Указатель (`const LogStyle*`) на один из общих объектов стиля.

### 3.3. Хранение и фабрика
Для стабильности указателей в C++ использован контейнер `std::map<LogLevel, LogStyle>`. В отличие от вектора, `std::map` гарантирует, что адреса объектов не изменятся при добавлении новых элементов, что исключает риск появления «битых» указателей.

## 4. Сравнение реализаций

### 4.1. NaiveEngine (Наивный подход)
Каждая запись `LogEntry` занимает фиксированную память структуры + динамическую память под две строки (`message` и `fontName`).
*   **Память:** $N \times (\text{sizeof(LogEntry)} + \text{length(message)} + \text{length(fontName)})$.
*   **Объекты:** Количество объектов стиля равно количеству строк лога.

### 4.2. FlyweightEngine (Оптимизированный подход)
Память расходуется только на уникальный текст и 8-байтовый указатель.
*   **Память:** $N \times (\text{sizeof(FlyWeightEntry)} + \text{length(message)}) + 4 \times \text{sizeof(LogStyle)}$.
*   **Объекты:** Количество объектов стиля константно (не более 4-х для всей системы).

---

## 5. Технологический стек и оптимизации

*   **Язык:** C++17. Выбор обусловлен необходимостью прямого контроля над памятью и эффективной работы с указателями.
*   **Графика:** GLFW + OpenGL3.
*   **UI:** Dear ImGui.
*   **Оптимизация рендеринга:** Использован `ImGuiListClipper`. Это позволяет приложению мгновенно отрисовывать миллионы строк, так как фактически обрабатываются только те записи, которые видны в данный момент в окне (20-50 строк из миллиона).
*   **Система сборки:** CMake с модулем `FetchContent` для автоматического управления зависимостями.

## 6. Результаты тестирования
В ходе демонстрации приложения были получены следующие результаты при генерации **1 000 000 (одного миллиона)** записей лога:

| Показатель | Naive Engine | Flyweight Engine | Экономия |
| :--- | :--- | :--- | :--- |
| **Количество объектов стиля** | 1 000 000 | 4 | **99.99%** |
| **Потребление RAM (примерное)** | ~280 MB | ~95 MB | **~66%** |

*Примечание: экономия памяти в 66% достигается даже на коротких сообщениях. При использовании более тяжелых метаданных (текстуры, иконки, сложные графы зависимостей) эффективность паттерна возрастает до 90-95%.*

## 7. Заключение
Проект наглядно демонстрирует необходимость паттерна «Легковес» в системах, работающих с большими объемами данных. Благодаря разделению состояний удалось снизить нагрузку на память в несколько раз и обеспечить стабильную работу приложения при экстремальных нагрузках. Архитектура проекта позволяет легко расширять типы логов и добавлять новые способы оптимизации без нарушения логики работы GUI.
